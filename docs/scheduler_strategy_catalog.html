ExpoNow Scheduler Strategy Catalog

This document compares the current custom heuristic with several well-known optimization approaches that fit an electric bus charging scheduler. It also explains how a strategy dropdown would wire into src/scheduler/strategies/ and into the Streamlit UI.

1. Current Implemented Strategy: Custom Heuristic

This is the engine that exists today in src/scheduler/strategies/custom_heuristic.py. It is greedy across buses: sort buses by departure time, enumerate feasible stop combinations for each bus, score the candidates using wait and travel time, commit the best one, then move to the next bus using the updated reservations.

How it works

  • Sort buses by (departure_minutes, id).
  • Generate route-order-preserving candidate stop sets.
  • For each candidate, probe charger availability through the reservation manager.
  • Choose the lowest-score candidate that still fits the current reservation state.
  • Commit the chosen slots and build the timeline events.

Complexity and fit

  • Candidate generation is exponential in the number of intermediate stations because it uses combinations.
  • Per candidate, the scheduler scans charger lanes and existing reservations to find slots.
  • Good fit for small to medium scenarios, deterministic runs, and explainable output.
  • Weak point: earlier buses can block later buses because the algorithm does not re-optimize globally.

2. OR-Tools CP-SAT

OR-Tools CP-SAT is the most natural exact upgrade for this project. The scheduler can model each bus stop as an optional interval, put each charger lane under a no-overlap constraint, and minimize wait or total journey time globally. Google documents CP-SAT as a constraint programming / integer optimization solver with strong support for scheduling primitives.

Pros

  • Native support for interval and resource constraints.
  • Can optimize all buses together instead of committing greedily.
  • Works well for hard constraints plus weighted soft objectives.
  • Good match for charger lane occupancy, waiting windows, and fixed route order.

Cons

  • Requires a rewrite of the scheduling model.
  • All times and durations must be integer-scaled.
  • Large instances may still need careful tuning and time limits.
  • Less transparent than the current hand-rolled greedy flow.
Complexity The underlying problem is still combinatorial and generally NP-hard. CP-SAT does not make that go away; it just gives the solver much better machinery for pruning and search.

3. Z3 Optimize

Z3 is excellent when the main concern is satisfiability, symbolic reasoning, or complex logical constraints. It can also optimize objectives, so it can solve scheduling-style formulations, but it is usually more natural for proof-heavy or rule-heavy problems than for rich schedule packing.

Pros

  • Very expressive for logical rules, implications, and soft constraints.
  • Useful for proving that a schedule exists or does not exist.
  • Strong fit for validation layers and rule experimentation.

Cons

  • Not as purpose-built for scheduling as CP-SAT.
  • Modeling time and solver time can be higher on operational schedules.
  • Usually less ergonomic for interval-style resource allocation.
Complexity Worst-case solving remains exponential, and in practice the usefulness depends on how the schedule is encoded. Z3 shines when logic is the main problem, not when the model is mostly temporal resource packing.
StartDeclare buses, stations, times, and boolean decision variables.
↓
Add logicEncode range rules, ordering rules, mutual exclusion, and soft penalties.
↓
OptimizeTell Z3 what to minimize or maximize.
↓
Check modelAsk whether the constraints are satisfiable and inspect any counterexample if not.
↓
Extract scheduleMap the model back into timeline events and reservations.

4. PuLP / MILP

PuLP is a modeling layer for linear and mixed-integer programming. It is a good fit if we want to write the scheduling problem as binary assignment and time-indexed constraints, then solve it with a MILP backend such as CBC, GLPK, CPLEX, or Gurobi.

Pros

  • Readable mathematical formulation.
  • Solver-agnostic: the model can be paired with multiple MILP backends.
  • Great for comparing objective tradeoffs and sensitivity to weights.

Cons

  • Time-expanded MILPs can become large very quickly.
  • Interval-style resource rules need extra binary variables or linearization.
  • Often less elegant than CP-SAT for pure schedule packing.
Complexity MILP is NP-hard in general. The formulation can still be practical for our problem size if the horizon is not too long and the decision grid stays compact.
StartChoose a time grid or binary decision structure.
↓
Define variablesCreate assignment, start-time, lane, and wait variables.
↓
Add linear constraintsEncode range, order, station capacity, and charging duration.
↓
Set objectiveMinimize the weighted schedule cost.
↓
Solve with backendPass the model to CBC, GLPK, or a commercial solver.
↓
Post-processTurn the selected binaries into a readable schedule.

5. Pyomo as a Modeling Layer

Pyomo is not a solver by itself. It is useful when the scheduling model grows beyond a single formulation or when we want to experiment with MILP, nonlinear, or generalized disjunctive programming models without rewriting the application each time.

Pros

  • Very flexible for advanced formulations and solver swapping.
  • Works well when the problem eventually expands into infrastructure planning plus operations.
  • Good if we want a model-first architecture.

Cons

  • More abstraction than PuLP.
  • Still requires a backend solver to do the actual optimization.
  • Can be overkill for the current project size.
StartDescribe the scheduling model in algebraic form.
↓

↓

↓
Solve and compareRun the model with multiple solvers if needed.
↓

6. Metaheuristics: ALNS, GA, SA, GRASP

Literature on electric bus scheduling regularly uses metaheuristics when the instance size grows or when exact methods become too slow. Adaptive large neighborhood search, genetic algorithms, simulated annealing, and GRASP are all well known in this space. They are especially useful for large fleets, rerouting under uncertainty, or hybrid models with many soft objectives.

Pros

  • Flexible and easy to hybridize with domain rules.
  • Scales better than exact solvers on some large instances.
  • Good for re-optimization and rolling-horizon scheduling.

Cons

  • No optimality guarantee.
  • Requires tuning of operators, temperature, or mutation settings.
  • Results can be less deterministic unless we control the random seed carefully.
Complexity The complexity is usually driven by the number of iterations, neighborhood moves, or restarts rather than by a closed-form worst-case bound. That makes these methods flexible, but also more dependent on tuning and benchmarking.
StartBuild an initial feasible schedule, often from a greedy heuristic.
↓
Destroy / mutateRemove, swap, or perturb part of the current solution.
↓
Repair / rebuildReinsert decisions using heuristics or local search.
↓
Accept or rejectKeep the change if it improves the score or passes a probabilistic rule.
↓
RepeatContinue until the time budget, iteration cap, or convergence condition is met.

7. Network Flow, Decomposition, and Hybrid Methods

These are not always the first tools people reach for, but they are very real options. If the scheduler can be represented as a network, min-cost flow can be effective. If the problem splits naturally into infrastructure, route assignment, and charging decisions, then Benders-style or other decomposition methods can be powerful.

Pros

  • Can be very strong when the problem structure matches the formulation.
  • Good candidate for scaling a larger system beyond a single-day schedule.
  • Allows one solver to handle a subproblem while another handles the master problem.

Cons

  • Design effort is higher than a direct exact or greedy approach.
  • Hybrid systems are more complex to test and debug.
  • The formulation can become very domain-specific.
StartSplit the problem into route, charging, and capacity decisions.
↓
Master problemDecide the high-level assignment or network flow structure.
↓
SubproblemOptimize charging times or lane occupancy for the current master solution.
↓
FeedbackReturn feasibility cuts or cost improvements to the master problem.
↓
ConvergeRepeat until no better solution appears.

How To Add Another Strategy

The app now keeps algorithms behind the shared SchedulerStrategy contract in src/scheduler/contract.py. The Streamlit page selects a scenario and a solver engine separately, then passes both ids into the view model so the selected strategy can produce the same ScheduleResult shape the reporting layer already understands.

Recommended folder shape

src/scheduler/
  contract.py
  candidates.py
  reservations.py
  scoring.py
  timeline.py
  results.py
  strategies/
    custom_heuristic.py
    cp_sat_strategy.py
    z3_strategy.py
    pulp_strategy.py
    registry.py

Integration steps

  • Create a new class that implements schedule(scenario) -> ScheduleResult.
  • Keep the strategy pure: no Streamlit imports, no UI code, no direct reporting imports.
  • Return the shared contract objects: BusPlan, StationReservation, ScheduleMetrics, and ScoreBreakdown.
  • Add the new class to src/scheduler/strategies/registry.py with an id, label, description, experimental flag, and availability check.
  • Let the UI call render_strategy_selector(); the registry controls which strategies are visible.
  • Run the shared invariant tests against every strategy so the UI never has to know solver internals.
Install notes Base app dependencies remain in requirements.txt. CP-SAT support is optional: install requirements-solver-cpsat.txt to make OR-Tools CP-SAT (experimental) appear in the strategy dropdown. Without OR-Tools installed, the registry hides CP-SAT from normal UI selection; direct calls return an unavailable-strategy warning.
UI wiring pattern The app should keep scenario selection and solver selection separate. The scenario dropdown picks the input world. The strategy dropdown picks the optimization engine. Then the result renderer consumes only ScheduleResult, which keeps the UI isolated from solver details.
Scenario selectedUser picks a scenario from the existing dropdown.
↓
Strategy selectedUser chooses a solver from the strategy dropdown.
↓
InstantiateLook up the strategy class in the registry and create the solver.
↓
ScheduleCall schedule(scenario) and receive ScheduleResult.
↓
RenderUse the existing reporting layer to display timelines, queues, and metrics.
