ExpoNow architecture guide · v2

A Bus Charging Scheduler Built for Clarity

ExpoNow is a Python + Streamlit application that schedules electric bus charging along a fixed inter-city route. This document describes the system as it stands: how the code is structured, why it is structured that way, and how each piece fits together.

What It Does, Briefly

A reviewer picks a scenario from a dropdown. The application loads route, station, and bus data from a JSON file, validates it, runs a scheduling engine, and displays the resulting bus timetables and station charging queues.

The scheduling problem is straightforward in concept and constrained in practice:

  • A fleet of buses travels a fixed route from Bengaluru to Kochi (or reverse), a 540 km corridor with four intermediate charging stations: A, B, C, and D.
  • Each bus departs at a given time and must never exceed 240 km between charges. Charging always takes exactly 25 minutes.
  • Each station has a limited number of charger lanes (currently 1 per station). Two buses cannot occupy the same lane simultaneously.
  • The scheduler must pick which stations each bus stops at, in what order, and assign charger reservations so that no constraints are violated. Among valid schedules, it prefers those that minimise total wait time and keep the fleet experience fair across operators.

Five assignment scenarios encode different departure patterns — evenly spaced, bunched starts, asymmetric fleet sizes, weighted operator fairness, and tight departure windows. Each is a first-class JSON document.

What the Architecture Forbids

Domain must not import Streamlit, adapters, or scheduler code. The domain is a pure model layer. If domain code ever references st., the layering contract is broken. A test scans every file in src/domain/ and fails on import streamlit or from streamlit.
Scheduler must not import Streamlit. The scheduler runs the core algorithm. It should be runnable in a terminal, a test, or a CI pipeline without a browser. A dedicated test (test_scheduler_package_does_not_import_streamlit) enforces this by grepping source files.
Reporting must not import Streamlit. The reporting layer produces data tables and diagram strings. Keeping it Streamlit-free means the same functions can power a CLI, an API response, or a PDF export without changes.

Domain Models: The Nouns of the Problem

Every entity in the scheduling problem is a frozen dataclass in src/domain/models.py. Frozen dataclasses are immutable — once constructed, they never change. This eliminates accidental mutation bugs and makes objects safe to share across threads.

Model Hierarchy

Scenario              # Root aggregate — everything belongs to one scenario
  schema_version      # int — for future format migrations
  id, name, description
  route               # Route
    name              # str
    stops             # list[str] — ["Bengaluru", "A", "B", "C", "D", "Kochi"]
    segments          # list[Segment] — each has from_stop, to_stop, distance_km
  stations            # list[Station] — each has id, charger_count
  buses               # list[Bus] — each has id, operator, direction, departure_minutes
  charging_policy     # ChargingPolicy — range_km (240), full_charge_minutes (25)
  travel_policy       # TravelPolicy — speed_kmph (60)
  weights             # Weights — individual, operator, overall (each defaults to 1.0)

The 1:1 mapping between JSON keys and domain types means the loader is a straightforward parse — not a complex transliteration layer.

Why Immutability Matters Here

  • No accidental mutation. A function receiving a Scenario cannot corrupt it for the caller. The scheduler reads the scenario; it never modifies it.
  • Hashable. Frozen dataclasses can serve as dictionary keys or live in sets.
  • Self-documenting. "This value never changes" is encoded in the type, not in a comment.
  • The trade-off — updating a frozen object requires constructing a new one. The scheduler produces one result per run, so the cost is negligible.

Scheduler Engine

The scheduler lives in src/scheduler/ and is the heart of the application. It implements a greedy heuristic: process buses in departure order, evaluate every feasible charging-stop combination for each bus, pick the one that scores best, and commit the required station reservations.

Architecture Within the Scheduler

The scheduler package is itself layered:

strategies/custom_heuristic.pyOrchestrates the per-bus scheduling loop. For each bus, generates candidates, probes for available slots, scores options, commits the best one, and builds the timeline.
candidates.pyGenerates all feasible charging-stop combinations for a given direction and range. Uses itertools.combinations over intermediate stations and discards combos where any gap exceeds the battery range.
constraints.pyValidates a completed schedule against hard rules: range never exceeded, route order preserved, no overlapping charger reservations, charge durations exact.
reservations.pyReservationManager — tracks per-lane occupied slots for each station. Provides find_slot() (probe without committing) and commit_slot() (irrevocable assignment).
scoring.pyComputes three weighted score components: individual wait (total bus wait minutes), operator smoothness (per-operator wait imbalance), and overall network (total fleet journey time). The score function registry is extensible by design.
contract.pyDefines SchedulerStrategy (a Protocol), ScheduleResult, and all output types: BusPlan, ChargingStop, TimelineEvent, StationReservation, ScheduleMetrics, ScoreBreakdown.

Hard constraints act as gates — candidates that fail them are discarded and never scored. Soft constraints rank the remaining candidates; the one with the lowest weighted score is selected. This two-phase approach guarantees that every output schedule satisfies all physical and operational limits while still optimising for fleet efficiency and fairness.

Station Reservation Management

The ReservationManager (src/scheduler/reservations.py) is a per-station, per-lane slot tracker:

  • find_slot(station, arrival, duration) — probes every lane at a station and returns the earliest available slot that fits after the bus arrives. Does not commit.
  • commit_slot(station, slot) — irrevocably books the slot. Subsequent calls are aware of it.
  • request(station, arrival, duration) — convenience that calls find then commit.

The scheduler uses find_slot during candidate evaluation (to score without mutating state) and commit_slot only for the chosen candidate. This probe-then-commit pattern is essential for correctness: if the scheduler modified state during evaluation, bus ordering would produce non-deterministic results.

The Strategy Protocol

SchedulerStrategy is a Protocol, not an abstract base class. Any class with a schedule(self, scenario) -> ScheduleResult method satisfies the contract automatically — no inheritance, no registration, no conditional imports:

class SchedulerStrategy(Protocol):
    def schedule(self, scenario: Scenario) -> ScheduleResult:
        ...

This design makes it trivial to swap in a different solver (OR-Tools, Z3, a deterministic algorithm). Each implementation lives in src/scheduler/strategies/ and is selected in app_view_model.py by changing one import.

Strategy Registry and Multi-Solver Architecture

ExpoNow v2 introduces a strategy registry that decouples solver selection from the application wiring. Users can switch between scheduling algorithms at runtime, and new solvers can be added without touching the UI, orchestration, or domain layers.

How the Registry Works

The registry (src/scheduler/strategies/registry.py) maps strategy IDs to option metadata and factory callables:

StrategyOptionFrozen dataclass with id, label, description, is_experimental, and is_available. Built at import time; the UI reads these fields to populate the solver dropdown.
_StrategyRegistrationPairs an option_factory (lazy metadata) with a strategy_factory (lazy class constructor). Strategies are never instantiated until get_strategy() is called.
list_strategy_options()Returns StrategyOption objects for every available strategy. Call with include_unavailable=True to inspect all registered strategies for diagnostics.
get_strategy(id)Constructs and returns a SchedulerStrategy instance. Raises ValueError with a clear message for unknown IDs.

The registry currently holds two strategies:

IDLabelExperimental?Availability
custom_heuristic Custom heuristic No - the default Always available
cp_sat OR-Tools CP-SAT Yes - optional dependency Available when ortools is installed

The SchedulerStrategy protocol is the sole contract. Every solver - greedy heuristic, CP-SAT, or a future algorithm - implements schedule(scenario) -> ScheduleResult. The orchestration layer never imports a specific strategy class; it only calls get_strategy() with the ID the user selected.

What the Registry Enables

  • No dead imports. CP-SAT imports ortools only inside the strategy class, not at module level. When OR-Tools is absent, the registry still reports the option as is_available=False - the code is importable, and the strategy politely returns a warning result.
  • Disjoint test surface. Each strategy has its own test file (test_cp_sat_strategy.py, test_strategy_registry.py). Integration tests loop over list_strategy_options() and verify that every available strategy produces valid results for core scenarios.
  • Zero UI changes when adding a solver. Register a new _StrategyRegistration, and the solver dropdown populates automatically. No layout or component code needs modification.
  • Guaranteed diagnostic path. list_strategy_options(include_unavailable=True) exposes every registered strategy - even those whose dependencies are missing - so operators and tests can report what could be available.

CP-SAT Strategy: Global Optimisation with Constraint Programming

The CP-SAT strategy (src/scheduler/strategies/cp_sat_strategy.py) replaces the greedy per-bus heuristic with a global constraint programming model solved by Google OR-Tools. Instead of making irrevocable decisions bus-by-bus, it encodes the entire fleet's constraints into a single mathematical model and lets the solver find a globally optimal assignment.

Greedy vs. Global: Why It Matters

Custom heuristic (greedy) Processes buses in departure order. Each bus picks its best candidate given previously committed reservations. Early buses can block later ones from optimal charging stops - a locally optimal choice may be globally suboptimal. Guarantees feasibility but not optimality.
CP-SAT (global) Formulates the entire problem as one constraint satisfaction / optimisation model. The solver explores the full decision space simultaneously - which buses charge where, in what order, on which lane - and minimises the weighted cost function. Can prove optimality or infeasibility, not just find a solution.

Neither approach is categorically better. The heuristic is fast, deterministic, and always available. CP-SAT trades runtime for proof of optimality and can escape local minima that trap the greedy algorithm. The registry lets users pick the right tool for the job at hand.

Decision Variables

The CP-SAT model creates one set of variables per bus, per candidate, per stop:

VariableTypePurpose
selectedBoolVarWhether this candidate is the chosen charging plan for the bus. Exactly one candidate per bus must be selected.
startIntVar(0, horizon)Minute at which charging begins at this stop. Must be ≥ arrival time when the candidate is selected.
waitIntVar(0, horizon)Idle minutes between arrival and charge start: wait = start - arrival. Contributes to the objective.
lane_selectorslist[BoolVar]One boolean per charger lane at the station. Exactly one lane is selected per stop when the candidate is active.
journeyIntVar(0, horizon)Total journey time for the bus: final_arrival - departure_minutes. Contributes to the objective.

Every variable is conditioned on selected using OnlyEnforceIf. When a candidate is not selected, its variables are forced to zero (wait, journey) or have their constraints deactivated (start, lane). This keeps the model sparse and avoids explosion from irrelevant alternatives.

Constraints: What the Solver Must Obey

  • One candidate per bus. sum(selected) == 1 across all candidates for each bus. Every bus must have exactly one charging plan.
  • Arrival sequencing. start ≥ arrival enforces that charging cannot begin before the bus reaches the station. Travel times are computed from segment distances and speed.
  • Wait accounting. wait = start - arrival captures idle time at each stop. When selected is false, wait is forced to 0.
  • Lane exclusivity. sum(lane_selectors) == 1 per stop (one lane per charge event). Across all buses, AddNoOverlap(intervals) per lane ensures no two buses occupy the same charger simultaneously. Each charging interval is modeled as an OptionalIntervalVar(start, duration, end, lane_selected).
  • Journey accounting. journey = (last_charge_end + final_travel) - departure for each candidate. Contributes to the overall network score.

Objective Function

The solver minimises a weighted sum of two terms that mirror the scoring system:

minimise: w_individual * sum(all wait terms)
        + w_overall * sum(all journey terms)

Weights come from the scenario's weights object and are scaled by a factor of 1000 to preserve integer arithmetic (CP-SAT uses integer coefficients). The operator smoothness component is not optimised directly by CP-SAT - it is computed post-hoc by finalize_schedule_result(), which both strategies share.

Solver Configuration and Time Budget

The solver is configured for a practical balance of speed and quality:

  • Time limit: max_time_in_seconds = 60. If the solver cannot prove optimality within one minute, it returns the best feasible solution found (marked FEASIBLE, not OPTIMAL).
  • Single worker: num_search_workers = 1. With 20 buses and up to dozens of candidates each, the model's decision space is large enough that deterministic single-threaded search is preferred over parallel exploration with non-deterministic results.
  • Horizon: computed as max_departure + route_time + serial_charge_time + 60 minutes. This upper bound is generous enough to accommodate any feasible schedule without making the integer variable domains unnecessarily large.

Shared Infrastructure Between Strategies

Both strategies reuse three modules that were extracted from the original custom_heuristic.py:

src/scheduler/timeline.pybuild_bus_timeline() - constructs a BusPlan with chronologically ordered TimelineEvent objects (departure, travel, arrival, wait, charge start/end, final arrival). compute_travel_minutes() - converts distance + speed to minutes.
src/scheduler/results.pyfinalize_schedule_result() - computes metrics, adds score breakdown, validates invariants, and returns the final ScheduleResult. compute_schedule_metrics() - aggregates total wait and max wait across all reservations.
src/scheduler/candidates.pygenerate_candidates() - enumerates all range-feasible charging-stop combinations using itertools.combinations. Both strategies call this function; CP-SAT uses the candidates as the decision variable domain.

This shared layer guarantees that both strategies produce identical output structures and undergo the same invariant validation. Adding a third solver means implementing schedule(scenario), calling generate_candidates() and finalize_schedule_result(), and registering it - the plumbing is done.

Scoring: How Good Schedules Are Ranked

Among all feasible schedules, some are better than others. The scoring system assigns a weighted numeric score to every ScheduleResult, letting the heuristic pick the candidate that balances individual wait, operator fairness, and network efficiency.

The Three Components

Individual WaitTotal wait minutes across all buses. A bus that waits 10 minutes at station A and 5 minutes at station B contributes 15 to this score. Lower is better.
Operator SmoothnessPenalises imbalance in average wait time across operators. If KPN buses average 8 minutes of wait and Freshbus averages 30, the score is the sum of absolute deviations from the fleet-wide mean. Lower means fairer.
Overall NetworkSum of every bus's total journey time (departure to final arrival). Captures network-level congestion. Lower means the fleet finishes faster.

Score Composition

Each component produces unweighted, weight, and weighted values. The total score is the sum of weighted values:

total_weighted = individual_weight * individual_unweighted
               + operator_weight * operator_unweighted
               + overall_weight * overall_unweighted

Weights live in the scenario JSON (weights.individual, weights.operator, weights.overall). Scenario 4 sets operator: 2.0 to test whether the scheduler responds to fairness pressure. The score functions are registered in a dictionary (SCORE_COMPONENTS) so adding a new dimension (e.g., peak_hour_penalty) means adding one function and one entry to the registry — no other code changes.

Entry Point and Orchestration

app.py is deliberately minimal. All wiring lives in src/app_view_model.py, which returns a frozen InitialViewModel dataclass containing everything the UI needs.

The View Model Pattern

app.py does three things:

def main():
    scenarios = list_scenario_summaries()
    selected_scenario_id = render_scenario_selector(scenarios)
    view_model = _get_cached_view_model(selected_scenario_id)
    render_input_view(view_model)
    if not view_model.validation_errors:
        render_schedule_output(view_model.schedule_result)

build_initial_view_model() does the heavy lifting: loads the scenario, validates it, runs the scheduler (if valid), and bundles everything into an InitialViewModel. This separation means:

  • The orchestration function is testable without a browser. A test can call build_initial_view_model("scenario_1") and inspect the returned InitialViewModel — no Streamlit runtime required.
  • The UI functions are pure rendering. They receive data and produce markup. No business logic, no data loading, no side effects.
  • Changes to the UI never break the data pipeline.

Reporting and UI

The reporting layer and UI layer have a clean division: reporting builds data structures, the UI renders them. Reporting knows nothing about Streamlit; the UI knows nothing about the domain models' internals.

Reporting Layer (src/reporting/)

tables.py transforms Scenario and ScheduleResult objects into lists of dicts — each one ready to become a table row:

  • build_summary_rows(scenario) — ID, name, description, distance, bus count
  • build_route_diagram_dot(scenario) — Graphviz DOT source for route visualisation
  • build_route_table(scenario) — per-segment from/to/distance
  • build_station_table(scenario) — station ID and charger count
  • build_bus_departure_table(scenario) — bus ID, operator, direction, departure time
  • build_bus_timetable_rows(result) — full per-stop charging schedule for every bus
  • build_station_queue_rows(result) — per-station charging order grouped by station

metrics.py builds summary metrics and score breakdown rows from ScheduleResult.

UI Layer (src/ui/)

components.py contains one Streamlit rendering function per visual section. Each function imports streamlit inside its body (never at module level):

  • render_scenario_summary_table, render_route_diagram, render_route_table, render_station_table, render_policy_table, render_weight_table, render_bus_departure_table — input display
  • render_validation_errors — error block when validation fails
  • render_bus_timetable, render_station_queues, render_schedule_metrics, render_score_breakdown — output display
  • render_infeasible_message — fallback when scheduling fails

layout.py orchestrates the rendering pipeline — calls reporting functions to get data, then calls component functions to render it.

Testing Strategy

The test suite uses Python's built-in unittest framework. No external test runner, no browser, no network. Run all tests with python -m unittest discover -s tests.

Test Categories

Domain and helpers
  • Time parsing: parse_hhmm and format_minutes round-trip and reject invalid input.
  • Route helpers: direction-aware stop ordering, total distance (540 km), segment distances, distance_between.
  • Domain models: construction, default values, frozen immutability.
  • Schedule contracts: BusPlan, ChargingStop, ScoreBreakdown, typed result.
Constraints and invariants
  • Range constraint: exact boundary allowed, over-range rejected, reverse direction.
  • Route order: valid direction, invalid direction, consecutive duplicate stop.
  • Complete schedule invariants: range violations, overlapping reservations, chronological order, charge duration.
  • Reservation manager: single charger serialisation, parallel lanes, gap-filling.
Scheduler integration
  • All five assignment scenarios produce feasible results with zero invariant violations.
  • Impossible scenarios (range too short) correctly produce infeasible results.
  • Score breakdown components are present and populated with correct weights.
  • Weight changes affect weighted total without changing hard feasibility.
  • Scenario 4's operator weight of 2.0 is properly threaded through.
Architecture enforcement
  • Scheduler package does not import Streamlit (grep-based test).
  • Orchestration imports without triggering Streamlit side effects.
  • Reporting package does not import Streamlit.
  • Validator accepts all five assignment scenarios, rejects malformed input.
  • Edge-case fixtures: exact range, impossible range, simultaneous arrivals, multi-operator.

Extensibility

The architecture is designed so that common future changes require data edits or small module additions — never rewrites.

Development Patterns You Will Find Here

Beyond the architecture, the codebase demonstrates several Python design patterns worth recognising. They appear across multiple files and are ready to reuse.

Frozen dataclasses as defaultEvery data type is @dataclass(frozen=True). Before adding a setter, ask "can this be immutable?" The answer is almost always yes.
Protocol over ABCSchedulerStrategy is a Protocol, not an abstract base class. Structural typing means implementations don't need to inherit from the base — they just need the right method signature.
Lazy Streamlit importsUI components import streamlit inside function bodies. The module is importable without a Streamlit runtime, which is essential for testing.
Grep-based architecture testsTests that scan source files for forbidden imports are cheaper and more reliable than code review comments. They run in CI and fail the build.
View model keeps app.py thinBusiness logic lives in build_initial_view_model(), not in app.py. The entry point orchestrates, it does not compute.
Explicit public APIPackage __init__.py files use __all__. Consumers import from the package (from src.domain import Scenario) rather than from specific modules.

In One Paragraph

ExpoNow is a layered Python application where the domain knows nothing about Streamlit, the scheduler knows nothing about I/O, and the UI knows nothing about scheduling logic. Data flows through explicit typed contracts: JSON → validated Scenario → scheduling algorithm → ScheduleResult → reporting tables → Streamlit widgets. Every layer is independently testable, and architecture boundaries are enforced by automated tests that scan for forbidden imports. The design prioritises clarity over cleverness — each file has one job, each function has one purpose, and the path from input to output is linear and inspectable at every step.

Layer Contracts

Every architectural boundary in ExpoNow is defined by an explicit contract: the data types that cross it, what the provider guarantees, and what the consumer can assume. These contracts are the seams at which each layer can be extracted, mocked, tested, and handed over independently.

Boundary 1: Adapters → Rest of System

The adapter layer is the only code that touches the file system or raw JSON. Its contract with the rest of the system is one-way: it delivers typed, validated domain objects.

Provider guarantees
  • load_scenario(id) always returns a valid Scenario or raises a typed AdapterError subclass — never a bare FileNotFoundError or json.JSONDecodeError.
  • validate_scenario(scenario) returns ALL validation errors in one pass, never fails fast after the first issue.
  • discover_scenario_ids() returns all scenario_*.json files found — the catalog is data-driven, not hardcoded.
  • No raw dict objects escape from this package. Every public function returns typed domain objects or plain list[str].
Consumer expectations
  • Consumers call load_scenario() with a valid scenario ID obtained from discover_scenario_ids() or list_scenario_summaries().
  • Consumers are responsible for calling validate_scenario() after loading, before scheduling.
  • Consumers handle ScenarioNotFoundError and MalformedScenarioError gracefully — typically by showing an error message in the UI.
Test seam: Mock the file system via a temp directory with known JSON fixtures. Test that every adapter function returns correct types. Test that malformed JSON, missing fields, and unknown scenario IDs produce the right typed errors. All five real scenario files must load and validate without errors.

Boundary 2: Scheduler → Rest of System

The scheduler is the algorithmic core. Its contract separates scheduling logic from everything else — the scheduler never touches I/O, never imports Streamlit, and never mutates the input scenario.

Provider guarantees
  • SchedulerStrategy.schedule(scenario) accepts a Scenario and always returns a ScheduleResult — even if the scenario is impossible (result is marked feasible=False with warnings).
  • The input Scenario is never modified. The scheduler is a pure function from Scenario → ScheduleResult.
  • Every BusPlan contains chronologically ordered TimelineEvent objects. Every StationReservation has non-overlapping time ranges per lane.
  • The result is validated against hard invariants via validate_schedule_invariants before being returned. Violations appear as warnings, not exceptions.
Consumer expectations
  • Consumers pass a fully validated Scenario. The scheduler does not re-validate domain data — it assumes the adapters layer has done its job.
  • Consumers check result.feasible before interpreting bus plans or metrics. An infeasible result has empty bus_plans and station_reservations.
  • Consumers read result.score_breakdown to understand schedule quality. The breakdown is None for infeasible results.
Test seam: The SchedulerStrategy protocol is the seam. Write a mock strategy that returns a known ScheduleResult to test downstream consumers (reporting, UI) without running the full algorithm. Test the real strategy against all five scenario files — each must produce a feasible result with zero invariant violations.

Boundary 3: Reporting → UI

Reporting is a pure transformation layer. It turns domain objects and scheduler results into display-friendly dicts. The UI never calls reporting functions with business logic intent — it calls them to get data for rendering.

Provider guarantees
  • Every function returns built-in Python types (list[dict], dict, str) — no custom classes, no dataclasses, no framework types.
  • Every dict has a consistent, documented set of keys. Consumers can rely on row["Field"], row["Value"], etc. being present.
  • All functions are pure: no I/O, no state, no Streamlit imports. The same input always produces the same output.
  • Functions accept only domain and scheduler contract types — never raw JSON or dict from the file system.
Consumer expectations
  • Consumers pass fully constructed Scenario and ScheduleResult objects. Reporting does not tolerate None for required fields.
  • Consumers treat the returned dicts as opaque display data — they format, order, style, and translate them without inspecting business meaning.
  • Consumers are free to iterate, filter, or augment the dicts for rendering purposes (e.g., adding Streamlit column config, colour coding).
Test seam: Reporting functions are the easiest to test — pure functions, no mocking needed. Test that each function returns the correct number of rows, that dict keys match the UI's expectations, and that edge cases (empty results, single-bus scenarios) produce valid output. A grep-based architecture test (test_reporting_package_does_not_import_streamlit) enforces the Streamlit ban.

Boundary 4: Orchestration → UI + Entry Point

The orchestration layer (app_view_model.py) is the single point where all data flows converge. It produces a frozen InitialViewModel dataclass that contains everything the UI needs — no lazy loading, no callbacks, no deferred computation.

Provider guarantees
  • build_initial_view_model(selected_scenario_id) always returns a fully populated InitialViewModel — even if loading, validation, or scheduling fails.
  • The InitialViewModel contains the loaded Scenario, all validation errors (if any), the full ScheduleResult (if scheduling ran), and the scenario summary list.
  • No business logic is deferred. When the view model is returned, every decision has been made — the UI only renders.
  • The function is deterministic for a given scenario ID (unless the JSON file changes).
Consumer expectations
  • app.py calls build_initial_view_model() on startup and caches the result (@st.cache_data). It never calls adapters or the scheduler directly.
  • src/ui/layout.py receives the InitialViewModel and destructures it. It checks view_model.validation_errors to decide whether to show schedule output.
  • UI functions never re-build domain objects, re-run validation, or re-schedule. They render what the view model gives them.
Test seam: build_initial_view_model() can be called without Streamlit — it returns a plain dataclass. Integration tests call this function with each of the five scenario IDs and verify that the returned InitialViewModel has no validation errors, a feasible schedule result, and populated metrics. This single function is the highest-value test in the suite.

Boundary 5: Domain → All Layers

The domain layer is the foundation. Every other layer depends on its types. The contract here is about what the domain does not do — it enforces the inward dependency rule that makes the architecture work.

Provider guarantees
  • All domain types are frozen (@dataclass(frozen=True)). Once constructed, they are immutable and hashable.
  • Domain types contain zero references to any other layer — no Streamlit, no scheduler, no adapters, no reporting. A grep for import streamlit across the entire src/domain/ directory must return zero matches.
  • Helper functions (parse_hhmm, format_minutes, distance_between, get_ordered_stops, total_distance, segment_distances) are pure and raise only standard Python exceptions (ValueError).
  • Default values exist for policies and weights (range_km=240, full_charge_minutes=25, speed_kmph=60, weights default to 1.0). A partial JSON can produce a valid scenario with sensible defaults.
Consumer obligations
  • Every layer constructs domain types only via their constructors — no subclassing, no monkey-patching, no __dict__ manipulation.
  • No layer stores domain objects in mutable containers that would allow accidental modification.
  • Every layer treats domain objects as opaque data carriers. Business logic processes them; it does not extend them.
Test seam: The domain layer is trivially testable — construct objects and verify field values, test helper functions with known inputs. An architecture-enforcement test scans every file in src/domain/ for forbidden imports and fails the build if any are found.
