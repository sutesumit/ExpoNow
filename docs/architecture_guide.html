Logic Flow

The scheduler has two nested logics: a bus must remain physically feasible, and each station must serialize charging demand through its limited charger capacity.

Constraints and Rules

Hard constraints decide whether a schedule is legal. Soft rules decide which legal schedule is preferred. Weights tune preference; they never excuse an invalid plan.

Hard constraints

  • Battery range must never exceed 240 km between full charges.
  • Buses visit route stations in order with no backtracking.
  • Each charger serves only one bus at a time.
  • Charging always takes exactly 25 minutes.
  • Bengaluru and Kochi are origin chargers, not scheduling stations.

Soft rules

  • Individual bus: avoid any single bus waiting too long.
  • Operator: keep each operator fleet smooth as a group.
  • Overall: keep total network time and congestion low.
  • Future rules can add pricing, priority, or driver constraints.

Implementation Toolkit: Reusable Patterns in This Codebase

Each section names a concept visible in Increment 0, explains the reasoning behind it, and shows exactly where to find it in the code. These patterns are ready to reuse in any Python project.

1. Frozen Dataclasses — Immutability as a Default

@dataclass(frozen=True) makes every field read-only after construction. Once built, a ScenarioSummary or ScheduleResult can never change. This seems restrictive, but it eliminates entire categories of bugs:

  • No accidental mutation — a function receiving a ScheduleResult cannot corrupt it for the caller.
  • Hashable by default — frozen dataclasses can be used as dictionary keys or stored in sets.
  • Thread-safe sharing — immutable objects are safe to pass between threads without locks.
  • Self-documenting contracts — "this value never changes" is encoded in the type, not in comments.

Where: src/domain/scenario.py:4-8, src/scheduler/contract.py:7-15, src/app_view_model.py:10-13

Trade-off: Updating a frozen object requires constructing a new one. For high-throughput mutation, this creates allocation pressure. The scheduler produces one result per run, so the trade-off is negligible here.

2. Protocol — Structural Typing Over Inheritance

SchedulerStrategy is a Protocol, not an abstract base class (ABC). Python's Protocol (PEP 544) checks that an object has the right shape — the right methods with the right signatures — rather than requiring an explicit inheritance chain.

  • No coupling to a base class — StubSchedulerStrategy never says extends SchedulerStrategy. It just has a schedule method with the right signature. The type checker verifies conformance automatically.
  • Zero-cost abstraction at runtime — Protocols are a type-checking-only construct; no runtime overhead.
  • Easier testing — a test can pass any object with a schedule(scenario) method. No need to mock or subclass.
  • Easier extension — a new solver (OR-Tools, Z3, deterministic) simply defines a class with the right method. No registry, no conditional imports.

Where: src/scheduler/contract.py:18-20 (Protocol definition), src/scheduler/stub.py:5-13 (implementation)

3. Ports-and-Adapters (Hexagonal Architecture)

The codebase is organized around a simple rule: dependencies point inward. The center is domain/, which imports nothing from the project. Every other layer may import from domain/, but domain/ never imports from them.

UI layerapp.py, src/ui/
↓
Reporting layersrc/reporting/
->
Adapters layersrc/adapters/
->
Domain layersrc/domain/, src/scheduler/

Why this matters:

  • Testability — the entire scheduler and domain can be tested without Streamlit. test_increment_0_harness.py proves this: it imports build_initial_view_model without ever touching st..
  • Swapability — replacing Streamlit with a CLI, REST API, or desktop app means rewriting only ui/ and app.py. Nothing else changes.
  • Separation of concerns — a developer debugging a scheduling bug opens scheduler/. A developer changing the UI layout opens ui/. Never both.
Forbidden: domain/ or scheduler/ imports streamlit. The test test_scheduler_package_does_not_import_streamlit enforces this automatically.

4. View Model Pattern — Orchestration vs. Rendering

app.py is deliberately thin — three lines of main():

build_initial_view_model(None) → render_scenario_selector(view_model) → build_initial_view_model(selected_id) → render_initial_view(view_model)

All the wiring lives in build_initial_view_model(), which returns an InitialViewModel — a frozen dataclass bundling everything the UI needs. This separation means:

  • The orchestration function is testable without a browser — test_app_orchestration_returns_selected_summary_and_result exercises the full data flow in pure Python.
  • The UI functions are pure rendering — they receive data and produce markup. No business logic, no data loading, no side effects.
  • Changes to the UI never break the data pipeline, and vice versa.

Where: src/app_view_model.py (orchestration), src/ui/layout.py (rendering)

5. Stub-Driven Development

StubSchedulerStrategy returns a fixed, feasible ScheduleResult with a warning that scheduling isn't implemented yet. This is a deliberate strategy:

  • The app runs end-to-end from day one — stakeholders see a working UI with scenario selection and placeholder results before any solver code is written.
  • Parallel work becomes possible — the UI developer and the scheduler developer can work concurrently. The UI is built against the stub; the scheduler is built against the contract.
  • The contract is validated early — test_stub_scheduler_returns_schedule_result_contract proves that ScheduleResult has the right shape. Any real scheduler must pass the same test.

Where: src/scheduler/stub.py:5-13

6. The Python Package Structure

The src/ directory is a Python package, not just a folder. Each subdirectory has an __init__.py:

  • src/__init__.py — docstring only: """ExpoNow application package."""
  • src/domain/__init__.py — re-exports ScenarioSummary via __all__
  • src/scheduler/__init__.py — re-exports ScheduleResult, SchedulerStrategy, StubSchedulerStrategy
  • src/adapters/__init__.py — docstring describing the package's role
  • src/ui/__init__.py — docstring describing the package's role
  • src/reporting/__init__.py — docstring describing the package's role

The __all__ pattern in domain and scheduler is intentional: it creates an explicit public API. Consumers import from the package (from src.domain import ScenarioSummary) rather than from specific modules (from src.domain.scenario import ScenarioSummary). This insulates consumers from internal file reorganizations.

Scenario Data: Shape, Rationale, and Test Coverage

Five JSON files in data/scenarios/ encode every assignment scenario as a self-contained, versioned document. The shape of each file was chosen to separate physical facts from operational policy from fleet schedules — so each dimension of the problem can evolve independently. The test suite proves the files match the assignment spec exactly and validates every structural invariant.

1. Scenario JSON Structure

Every scenario file follows the same skeleton. Only the buses list and weights object vary per scenario; everything else is shared constants from the assignment spec.

Top-level keys

{
  "schema_version": 1,
  "id":            "scenario_N",
  "name":          "Short title",
  "description":   "Intent of this scenario",
  "route":         { … },
  "stations":      [ … ],
  "buses":         [ … ],
  "charging_policy": { … },
  "travel_policy":   { … },
  "weights":       { … }
}

Each key's concern

schema_version   → file format evolution
id, name, desc   → discoverable scenario identity
route            → static world geography
stations         → physical infrastructure
buses            → dynamic fleet schedule
charging_policy  → battery physics
travel_policy    → movement physics
weights          → tunable optimisation

No nested magic numbers. Every physical constant (range, charge time, speed) lives under a named policy key. This makes changing any constant a one-field edit, not a hunt through code.

2. Why This Shape?

The shape maps directly to three independent axes of change, so that a change in one never requires a change in the others:

Physical worldroute, stations, charging_policy, travel_policy — the laws of this world. Change when the route gains a stop, a station gets a second charger, or battery technology improves range.
Fleet schedulebuses — the dynamic input. Every scenario has a different departure sheet. Adding a new scenario means creating a new file; the schema never changes.
Optimisation weightsweights — the tunable knobs. Scenario 4 overrides operator to 2.0 while leaving the rest at 1.0. No code changes, no new files, just a single number.

Design decisions in detail:

Decision
Why
Future-proofs
schema_version at the root
If the format evolves, old files are still readable. A loader can branch on version.
Backward-compatible schema migrations without breaking existing scenarios.
Segments as explicit from/to/distance_km objects
Route order is derived from the sequence, not hardcoded as "stop A is index 1." Any segment can be inserted or removed.

stations is a list of objects, not a flat array of IDs
Each station carries its own charger_count. Default is 1, but the field is already present — no schema change needed for multi-charger stations.
Scaling charger counts per station is data-only, engine-preserving.

New optimisation dimensions do not touch the fleet schedule or the physical world data.
buses stores operator as a string, not an enum or foreign key
Operators are facts about each bus trip, not a separately managed entity. Adding a new operator means writing its name in the data, not registering it in code.

Anti-pattern avoided: hardcoding scenario data in Python dictionaries inside src/adapters/scenario_catalog.py. 4. Test Coverage: Scenario Data Validation

tests/test_increment_1_scenario_data.py contains seven tests that prove the JSON files are structurally correct and match the assignment specification exactly. These tests run with Python's built-in unittest, no external dependencies, and no Streamlit import.

Existence & shape tests

  • All files present — asserts every scenario_{1..5}.json exists under data/scenarios/.
  • Required top-level keys — verifies each file contains schema_version, id, name, description, route, stations, buses, charging_policy, travel_policy, weights.
  • Bus row fields — every bus in every scenario has id, operator, direction, departure.

Assignment spec tests

  • Bus counts per scenario — scenario_3 has 14 buses (asymmetric); all others have 20.
  • Scenario 4 operator weight — asserts weights.operator == 2.0, the only scenario with a non-default weight.
  • Scenario 5 departure window — verifies all 20 departures fit within 72 minutes (19:00 to 20:12).
  • Station & policy defaults — validates stations are A–D with charger_count=1, range_km=240, full_charge_minutes=25, speed_kmph=60, segment distances match [100, 120, 100, 120, 100].
What these tests protect: If someone accidentally edits scenario_3 to have 20 buses, or changes scenario_4's operator weight back to 1.0, or introduces a typo in a departure time — the test suite catches it. These tests serve as the automated gate between "data that looks right" and "data that provably matches the assignment."

Coverage boundaries: These tests are not exhaustive of the scheduler's needs. They validate static properties of the input data. Once the scheduler engine exists, a second layer of tests will validate dynamic properties — e.g. "every feasible schedule respects the 240 km range rule" and "no two buses occupy the same charger at the same time." The scenario data tests are the foundation: they guarantee the scheduler receives a structurally valid world description.

2. Evolution: Before and After Increment 2

The architecture gained real types, a file loader, and a weight-aware stub. The viewer still sees the same dropdown — but internally every layer now passes typed domain objects.

Before (Increment 0 & 1)

  • Only ScenarioSummary(id, name, desc, is_placeholder=True) existed.
  • Hardcoded catalog in scenario_catalog.py returned placeholders.
  • SchedulerStrategy.schedule() accepted ScenarioSummary — no real data reached the scheduler.
  • ScheduleResult fields were list[Any] / dict[str, Any] — no type safety.

The data path from scheduler output to visible timetable follows six steps through three layers:

  1. CustomHeuristicStrategy.schedule() produces ScheduleResult with bus_plans (each containing typed TimelineEvents) and station_reservations (each with station, bus_id, charger_lane, start, end).

2. The Three Score Components

Scoring only runs on schedules that have already passed hard invariant validation. Every component is a penalty (lower is better). Each exposes an unweighted (raw) value and a weighted value (unweighted × weight).

1. Individual Wait

Raw: sum of all bus wait minutes across every charging stop.
Weighted: raw × weights.individual
Source: result.metrics.total_wait_minutes
Purpose: penalizes total time all buses collectively spend waiting for chargers. A high individual wait means heavy station contention.

2. Operator Smoothness

Raw: sum over operators of abs(avg_wait_per_operator − fleet_avg_wait)
Weighted: raw × weights.operator
Source: computed from per-operator wait stats extracted from BusPlan events and StationReservation data.
Purpose: penalizes imbalance where one operator's buses wait disproportionately longer than others. Higher value = less fair across operators.

3. Overall Network

Raw: sum of final_arrival_minutes − departure_minutes for every bus in the fleet.
Weighted: raw × weights.overall
Source: extracted from each BusPlan's first departure event and final_arrival_minutes.
Purpose: penalizes schedules that keep buses on the road longer overall. Encourages compact itineraries.

Total weighted score: total_weighted = individual_wait.weighted + operator_smoothness.weighted + overall_network.weighted
All three components are additive penalties. A lower total is always better.

3. How Weights Influence Candidate Selection

Weights affect scheduling at two levels: per-bus candidate ranking and global score reporting. The operator smoothness component is special — it can only be computed globally, not per-candidate.

Per-bus candidate scoring (at selection time)

When the heuristic evaluates candidates for a single bus, it scores each feasible candidate and picks the lowest score:

per_bus_score = weights.individual × candidate_wait + weights.overall × candidate_travel_time

candidate_wait = total minutes this bus would wait across all its charging stops.
candidate_travel_time = final_arrival − departure_minutes for this candidate.
Operator smoothness is excluded here — it requires knowing all buses' waits, which isn't available until after scheduling completes.

Global post-hoc scoring (after all buses scheduled)

After every bus has been assigned a candidate and reservations are committed, compute_score_breakdown() runs across the full ScheduleResult and produces the final ScoreBreakdown with all three components. This is what the UI displays and what tests assert against.

Consequence: changing weights.individual or weights.overall in a scenario file can change which candidate is selected for a bus — and therefore change the entire schedule. Changing weights.operator only affects the displayed score, not candidate selection (v1 limitation — documented below).

3. Soft Factor Scoring — From Candidates to Ranking

After the hard gate, every surviving candidate is a physically possible charging plan. Soft scoring ranks them so the heuristic can pick the best one:

Per-bus scoring formula (candidate selection)
score(c) = wi × wait_penalty(c) + wo × travel_penalty(c)
  • wi = weights.individual (default 1.0)
  • wait_penalty(c) = total minutes this bus would wait across all charging stops in candidate c
  • wo = weights.overall (default 1.0)
  • travel_penalty(c) = candidate's total trip time (final_arrival − departure)
Post-hoc global formula (after all buses committed)
total = wi × Σwait + wop × Σ|avg_op − fleet_avg| + wo × Σjourney
  • Σwait = sum of all bus wait minutes across the fleet
  • wop = weights.operator (default 1.0; 2.0 in Scenario 4)
  • Σ|avg_op − fleet_avg| = operator smoothness penalty
  • Σjourney = sum of all bus journey times
Weight sensitivity: Increasing weights.individual makes the algorithm favour candidates with less waiting, potentially at the cost of longer travel. Increasing weights.overall favours shorter total trip time, potentially at the cost of more waiting. Both can change which candidate is selected.

8. Hard + Soft → Output: Complete Formula Reference

The entire decision pipeline from input to output, expressed as a single compound formula:

Phase 1 — Candidate generation (hard gate):
C = {c ⊆ P(stations) \ {∅} | ∀ leg(c): distance(leg) ≤ range_km}

Phase 2 — Per-bus probe (hard gate):
C* = {c ∈ C | ∀ s ∈ c: find_slot(s, arrival(s), 25) ≠ None}

Phase 3 — Soft scoring (if |C*| > 0):
score(c) = wi × Σwait(s, c) + wo × (final_arrival(c) − departure)
chosen = argmin score(c) for c ∈ C*

Phase 4 — Global post-hoc (after all buses committed):
individual_wait = wi × Σb total_wait(b)
operator_smoothness = wop × Σop |avg_wait(op) − fleet_avg_wait|
overall_network = wo × Σb (final_arrival(b) − departure(b))
total_weighted = individual_wait + operator_smoothness + overall_network

Phase 5 — Invariant self-check:
violations = validate_schedule_invariants(scenario, result)
feasible = (|C*| = |buses| ∧ |violations| = 0)
P(stations) = power set of intermediate stations wi, wop, wo = weights from scenario JSON find_slot = ReservationManager lane probe |violations| = 0 = all 6 invariant checks pass

def generate_candidates(route, direction, range):

    # filter out terminals (Bengaluru, Kochi)

_find_earliest_slot() algorithm 
def _find_earliest_slot(existing, arrival, duration):
    cursor = arrival
    for (start, end) in existing:
        if start >= cursor + duration:
            break  # gap fits before this reservation
        cursor = max(cursor, end)
    if cursor + duration ≤ MAX_MINUTES:
        return LaneSlot(lane, cursor, cursor + duration)
    return None

Scans sorted reservations linearly, advances cursor past each overlapping interval. Returns the first gap where duration fits.

17. Key Architecture Rules in One Place

Allowed dependencies
  • adapters → domain — adapters import typed domain objects
  • scheduler → domain — scheduler imports Scenario + route utilities
  • scheduler → scheduler — intra-scheduler imports are free
  • reporting → domain, scheduler.contract — tables import models + contract types
  • ui → reporting, app_view_model, scheduler.contract — UI imports pre-computed data
  • app_view_model → adapters, scheduler, domain — orchestration wires all layers
Forbidden dependencies (test-enforced)
  • scheduler → streamlit — enforced by grep-based test
  • reporting → streamlit — enforced by grep-based test
  • domain → anything — domain imports only stdlib
  • ui → scheduler.internal — UI must not import scheduler internals (contract only)
  • scheduler → adapters — scheduler never reads files directly
The single most important rule: If scheduling code needs st., the boundary has already been broken. The scheduler, domain, adapters, and reporting layers are all independently testable without Streamlit. Only src/ui/ and app.py import Streamlit.
